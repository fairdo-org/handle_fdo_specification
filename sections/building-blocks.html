PID & Resolution

[H1-1] PID System
Each Handle-based FDO has a Handle identifier as its PID, assigned through the Handle System.
[H1-2] PID Detection
The syntax of a Handle identifier follows the regular expression: `\d+(\.\d+)*\/\S+`
[H1-3] PID Resolution
The resolution of a Handle-based FDO, which is identified by PID X, into an FDO record uses the global Handle Resolver to receive the Handle record at `https://hdl.handle.net/api/handles/[X]`.

Attribute mapping

All mandatory and optional attributes that are given in the FDO spec are implemented as Handles. Handle-based FDOs make direct use of these attributes. Hence, no additional mappings are required.

FDO records

This section specifies how an FDO record is implemented by Handle-based FDOs.
Handle record
Each Handle record consists of a list of Handle values. Each Handle value is represented as a JSON object with (at least) five types (see also Handle Technical Manual):
  • index (integer)
  • type (string)
  • data (object)
  • ttl (the time-to-live in seconds of the value, an integer (or, in the rare case of an absolute expiration time, that expiration time as an ISO8601-formatted string))
  • timestamp (an ISO8601-formatted string)
FDO record
The conceptional FDO record (according to the FDO specification) consists of tuples `(k_1:v_1),(k_2:v_2),…,(k_n:v_n)`.
[H2-1] Mapping the FDO Record to the Handle Record
We assume an FDO record which consists of FDO attributes/tuples `a_1=(k_1:v_1),a_2=(k_2:v_2),…, a_n=(k_n:v_n)`. Mapping the FDO record to a Handle record means to map all FDO attributes `a_1,...,a_n` to a list of Handle values `h_1,...,h_n` as follows: Each attribute `a_i` is assigned a positive integer `f_i`, where `f_1, ..., f_n` are mutually different. Assigning integers to tuples should take conventions from Handle identifiers into account (e.g., if `k_i=HS_ADMIN`, then `f_i=100`). Then, the FDO attribute `a_i` is mapped to the Handle value `h_i` by setting:
  • `index: f_i`
  • `type: k_i`
  • `data.value: v_i`
The remaining types of Handle value `h_i` are usually set according to their default values, but possibly also with different values, if this makes sense in the given context.
  • `data.format: string` (and `data.format: admin`, if `k_i=HS_ADMIN`)
  • `ttl: 86400`
  • `timestamp: [current time]`
As an example, the FDO record with handle identifier `123/abcde` containing the three FDO attributes
  • `(0.FDO/Type: 123/45678)`
  • `(0.FDO/Profile: 123/00001)`
  • `(HS_ADMIN: {"handle": "0.NA/21.T11971", "index": 200, "permissions": "111111111111"})`
is mapped to the following Handle record in JSON format: 
{
    "responseCode": 1,
    "handle": "123/abcde",
    "values": [
        {
            "index": 1,
            "type": "0.FDO/Type",
            "data": {
                "format": "string",
                "value": "123/45678"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 2,
            "type": "0.FDO/Profile",
            "data": {
                "format": "string",
                "value": "123/00001"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 100,
            "type": "HS_ADMIN",
            "data": {
                "format": "admin",
                "value": {
                    "handle": "0.NA/21.T11971",
                    "index": 200,
                    "permissions": "111111111111"
                }
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        }
    ]
}

[H2-2] Mapping the Handle Record to the FDO Record
Mapping a handle record with handle values `h_1,...,h_n` to an FDO record with attributes `a_1,...,a_n` means to map each handle value `h_i` to an FDO attribute `a_i`. The handle value `h_i` 
{
    "index": f_i,
    "type": "k_i",
    "data": {
        "format": "string",
        "value": "v_i"
    },
    "ttl": 86400,
    "timestamp": "2025-05-06T12:30:44Z"
}
is mapped to the FDO attribute `a_i=(k_i, v_i)`. All other types (`index`, `data.value`, `ttl`, `timestamp`) are dropped.

FDO profiles

[H3-1] Representation of FDO Profiles
The Handle record of a Handle-based FDO profile must contain a fixed, predefined set of FDO attributes (see Tables 1 and 2). Note that there are two types of attributes in the Handle record:
  1. Meta information about the profile, to describe the FDO profile itself: `0.FDO/Type`, `0.FDO/Profile`, `0.FDO/Name` (all mandatory), `0.FDO/Description` (optional).
  2. The actual content of the profile, from which all validation rules (for FDOs instantiating this profile) can be derived. There are two ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute `0.FDO/Data`. For more details, see Table 1.
    • Way 2: The FDO profile is fully specified in the Handle record, simply relying on references to other attributes (`0.FDO/Attribute`), potential profile extension (`0.FDO/Extends`) and a boolean (`0.FDO/denyAdditionalAttributes`). For more details, see Table 2.
    • Note that both ways can be combined by using all attributes `0.FDO/Data`, `0.FDO/Attribute`, `0.FDO/Extends`, `0.FDO/denyAdditionalAttributes` simultaneously. In this case, when creating or updating this FDO profile, it must be checked that both ways induce the same validation rules.

0.FDO/Profile The string 0.FDO/ProfileDef. Additional profiles may be added, if applicable. 1..*
0.FDO/Name Human-readable and localized label(s) for the profile. 1..*
0.FDO/Description Human-readable and localized description(s) for the profile. 0..*

FDO attribute definitions

[H5-1] Representation of FDO attribute definitions
The Handle record of a Handle-based FDO attribute definition must contain a fixed, predefined set of FDO attributes (see Tables 3 and 4). Note that there are two types of attributes in the Handle record:
  1. Meta information about the attribute definition, to describe the FDO attribute definition itself: `0.FDO/Type`, `0.FDO/Profile`, `0.FDO/Name`, `0.FDO/Cardinality` (all mandatory), `0.FDO/Description` (optional).
  2. The actual content of the attribute definition, from which all validation rules (for attributes instantiating this attribute definition) can be derived. There are two ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute `0.FDO/Data`. For more details, see Table 3.
    • Way 2: The FDO attribute definition is fully specified in the Handle record, simply relying on references to other attribute definitions, syntax definitions, or profiles (`0.FDO/DataType`) and a specified validation mechanism (`0.FDO/ValidationMechanism`). For more details, see Table 4.
    • Note that both ways can be combined by using all attributes `0.FDO/Data`, `0.FDO/DataType`, `0.FDO/ValidationMechanism` simultaneously. In this case, when creating or updating this FDO attribute definition, it must be checked that both ways induce the same validation rules.

[H5-1a] Encoding Semantics
A name must be and a description may be included in a Handle-based attribute definition using the attributes `0.FDO/Name` respectively `0.FDO/Description` in the FDO record.
  • Way 1: The name and the description can be repeated in the JSON schema using the keywords `title` and `description`.
[H5-1b] Encoding Syntax
  • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute `0.FDO/Data`. The JSON Schema encodes the following basic validation rules:
    1. An attribute states to follow one of the primitive data types “string”, “number”, “integer” and “boolean” using the `type` keyword.
    2. An attribute states to comply with a regular expression using the `pattern` keyword.
    3. An attribute states to lie within a numeric range using the keywords `minimum`, `exclusiveMinimum`, `maximum` or `exclusiveMaximum`.
    4. An attribute states to be equal to one element from a whitelist through the `enum` keyword.
    5. An attribute states to be not equal to all elements from a blacklist through combining `not` with `enum`. 
    {
    "$schema": "http://json-schema.org/draft-04/schema#",
    "title": "publication year",
    "description": "The publication year of the resource.",
    "type": "integer",
    "minimum": 0,
    "maximum": 2026
}
    {
    "$schema": "http://json-schema.org/draft-04/schema#",
    "title": "color",
    "description": "The color of the resource.",
    "type": "string",
    "enum": ["red", "yellow", "green", "blue"],
    "not":
    {
        "enum": ["black", "white"]
    }
}
  • Way 2: To specify the syntax of an attribute, the following two attributes must be used in the FDO record:
    1. `(0.FDO/ValidationMechanism, Syntax)`
    2. ` (0.FDO/DataType, [reference to a syntax definition])`.
    Syntax definitions are themselves FDOs and encode the basic validation rules in their records. They follow the syntax definition profile with the PID `0.FDO/SyntaxDef` and contain the attributes as in table 5.

[H5-1c] Complex attributes
  • Way 1:
    1. A composition of attributes is implemented in JSON schema as an object with properties. Each property links to the JSON schema of another attribute definitions using the keyword `$ref`. You can use the `required` keyword to make attributes within the composition mandatory and `additionalProperties` to prevent adding additional content to the composition. 
      {
  "$schema": "http://json-schema.org/draft-04/schema#",
  "title": "author",
  "description": "An author is a composition of a first name and a last name.",
  "type": "object",
  "properties": {
    "firstName": {
      "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/3df63b7acb0522da685d"
    },
    "lastName": {
      "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/3df63b7acb0522da685d"
    }
  },
  "required": ["firstName", "lastName"],
  "additionalProperties": false
}
      This attribute composition validates: 
      {
     "firstName": "Jon",
     "lastName": "Doe"
}
    2. A union of attributes is implemented by combining the keywords `oneof` and `$ref`to reference multiple existing JSON schemas, each representing an attribute definition. 
      {
  "$schema": "http://json-schema.org/draft-04/schema#",
  "title": "Union of string and integer",
  "description": "This attribute validates all strings and all integers.",
  "anyOf": [
    {
      "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/3df63b7acb0522da685d"
    },
    {
      "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/9edcffff4461c9083fbf"
    }
  ]
}
      This attribute union validates: 
      123
      and 
      "myString"
  • Way 2:
    1. A composition of attributes is implemented by setting two attributes in the FDO record:
      1. `(0.FDO/ValidationMechanism, Composition)`
      2. ` (0.FDO/DataType, [reference to attribute definition])`, repeated multiple times.
      The following three attributes define a composition of a string and an integer.
      1. `(0.FDO/ValidationMechanism, Composition)`,
      2. ` (0.FDO/DataType, 21.T11969/3df63b7acb0522da685d)`,
      3. ` (0.FDO/DataType, 21.T11969/9edcffff4461c9083fbf)`,
      The composition is instantiated by the following JSON object (String and Integer are the names of the attributes -> we need to define better examples here): 
      {"String": "abc", "Integer": 123}

      2. ` (0.FDO/DataType, [reference to attribute definition])`, repeated multiple times.
