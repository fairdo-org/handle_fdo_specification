PID & Resolution

[H1-1] PID System
Each Handle-based FDO has a Handle identifier as its PID, assigned through the Handle System.
[H1-2] PID Detection
The syntax of a Handle identifier follows the regular expression: `\d+(\.\d+)*\/\S+`
[H1-3] PID Resolution
The resolution of a Handle-based FDO, which is identified by PID X, into an FDO record uses the global Handle Resolver to receive the Handle record at `https://hdl.handle.net/api/handles/[X]`.

Attribute mapping

All mandatory and optional attributes that are given in the FDO spec are implemented as Handles. Handle-based FDOs make direct use of these attributes. Hence, no additional mappings are required.

FDO records

This section specifies how an FDO record is implemented by Handle-based FDOs.
Handle record
Each Handle record consists of a list of Handle values. Each Handle value is represented as a JSON object with (at least) five types (see also Handle Technical Manual):
  • index (integer)
  • type (string)
  • data (object)
  • ttl (the time-to-live in seconds of the value, an integer (or, in the rare case of an absolute expiration time, that expiration time as an ISO8601-formatted string))
  • timestamp (an ISO8601-formatted string)
FDO record
The conceptional FDO record (according to the FDO specification) consists of tuples `(k_1:v_1),(k_2:v_2),…,(k_n:v_n)`.
[H2-1] Mapping the FDO Record to the Handle Record
We assume an FDO record which consists of FDO attributes/tuples `a_1=(k_1:v_1),a_2=(k_2:v_2),…, a_n=(k_n:v_n)`. Mapping the FDO record to a Handle record means to map all FDO attributes `a_1,...,a_n` to a list of Handle values `h_1,...,h_n` as follows: Each attribute `a_i` is assigned a positive integer `f_i`, where `f_1, ..., f_n` are mutually different. Assigning integers to tuples should take conventions from Handle identifiers into account (e.g., if `k_i=HS_ADMIN`, then `f_i=100`). Then, the FDO attribute `a_i` is mapped to the Handle value `h_i` by setting:
  • `index: f_i`
  • `type: k_i`
  • `data.value: v_i`
The remaining types of Handle value `h_i` are usually set according to their default values, but possibly also with different values, if this makes sense in the given context.
  • `data.format: string` (and `data.format: admin`, if `k_i=HS_ADMIN`)
  • `ttl: 86400`
  • `timestamp: [current time]`
As an example, the FDO record with handle identifier `123/abcde` containing the three FDO attributes
  • `(0.FDO/Type: 123/45678)`
  • `(0.FDO/Profile: 123/00001)`
  • `(HS_ADMIN: {"handle": "0.NA/21.T11971", "index": 200, "permissions": "111111111111"})`
is mapped to the following Handle record in JSON format: 
{
    "responseCode": 1,
    "handle": "123/abcde",
    "values": [
        {
            "index": 1,
            "type": "0.FDO/Type",
            "data": {
                "format": "string",
                "value": "123/45678"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 2,
            "type": "0.FDO/Profile",
            "data": {
                "format": "string",
                "value": "123/00001"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 100,
            "type": "HS_ADMIN",
            "data": {
                "format": "admin",
                "value": {
                    "handle": "0.NA/21.T11971",
                    "index": 200,
                    "permissions": "111111111111"
                }
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        }
    ]
}

[H2-2] Mapping the Handle Record to the FDO Record
Mapping a handle record with handle values `h_1,...,h_n` to an FDO record with attributes `a_1,...,a_n` means to map each handle value `h_i` to an FDO attribute `a_i`. The handle value `h_i` 
{
    "index": f_i,
    "type": "k_i",
    "data": {
        "format": "string",
        "value": "v_i"
    },
    "ttl": 86400,
    "timestamp": "2025-05-06T12:30:44Z"
}
is mapped to the FDO attribute `a_i=(k_i, v_i)`. All other types (`index`, `data.value`, `ttl`, `timestamp`) are dropped.

FDO profiles

[H3-1] Representation of FDO Profiles
The Handle record of a Handle-based FDO profile must contain a fixed, predefined set of FDO attributes (see Tables 1 and 2). Note that there are two types of attributes in the Handle record:
  1. Meta information about the profile, to describe the FDO profile itself: `0.FDO/Type`, `0.FDO/Profile`, `0.FDO/Name` (all mandatory), `0.FDO/Description` (optional).
  2. The actual content of the profile, from which all validation rules (for FDOs instantiating this profile) can be derived. There are two ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute `0.FDO/Data`. For more details, see Table 1.
    • Way 2: The FDO profile is fully specified in the Handle record, simply relying on references to other attributes (`0.FDO/Attribute`), potential profile extension (`0.FDO/Extends`) and a boolean (`0.FDO/denyAdditionalAttributes`). For more details, see Table 2.
    • Note that both ways can be combined by using all attributes `0.FDO/Data`, `0.FDO/Attribute`, `0.FDO/Extends`, `0.FDO/denyAdditionalAttributes` simultaneously. In this case, when creating or updating this FDO profile, it must be checked that both ways induce the same validation rules.

FDO attribute definitions

[H5-1] Representation of FDO attribute definitions
The Handle record of a Handle-based FDO attribute definition must contain a fixed, predefined set of FDO attributes (see Tables 3 and 4). Note that there are two types of attributes in the Handle record:
  1. Meta information about the attribute definition, to describe the FDO attribute definition itself: `0.FDO/Type`, `0.FDO/Profile`, `0.FDO/Name`, `0.FDO/Cardinality` (all mandatory), `0.FDO/Description` (optional).

[H5-1a] Encoding Semantics
A name must be and a description may be included in a Handle-based attribute definition using the attributes `0.FDO/Name` respectively `0.FDO/Description` in the FDO record. If way 1 is used, the name and the description can be repeated in the JSON schema using the keywords `title` and `description`.
[H5-1b] Encoding Syntax
  • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute `0.FDO/Data`. The JSON Schema encodes the following basic validation rules:
    1. An attribute states to follow one of the primitive data types “string”, “number”, “integer” and “boolean” using the `type` keyword.
    2. An attribute states to comply with a regular expression using the `pattern` keyword.
    3. An attribute states to lie within a numeric range using the keywords `minimum`, `exclusiveMinimum`, `maximum` or `exclusiveMaximum`.
    4. An attribute states to be equal to one element from a whitelist through the `enum` keyword.
    5. An attribute states to be not equal to all elements from a blacklist through combining `not` with `enum`. 
    {
    "$schema": "http://json-schema.org/draft-04/schema#",
    "title": "publication year",
    "description": "The publication year of the resource.",
    "type": "integer",
    "minimum": 0,
    "maximum": 2026
}
    {
    "$schema": "http://json-schema.org/draft-04/schema#",
    "title": "color",
    "description": "The color of the resource.",
    "type": "string",
    "enum": ["red", "yellow", "green", "blue"],
    "not":
    {
        "enum": ["black", "white"]
    }
}
  • Way 2: To specify the syntax of an attribute, the following two attributes must be used in the FDO record:
    1. `(0.FDO/ValidationMechanism, Syntax)`
    2. ` (0.FDO/DataType, [reference to a syntax definition])`.
    Syntax definitions are themselves FDOs and encode the basic validation rules in their records. They follow the syntax definition profile with the PID `0.FDO/SyntaxDef` and contain the attributes as in table 5.

[H5-1e] Profile Referencing
Profile referencing permits an attribute definition A to specify that the value aval (of an attribute a=(adef,aval) that instantiates A) MUST be a reference to an FDO following a specific profile. Hence, it is a mechanism to enforce that specific attributes in FDO records reference other FDOs that follow a certain profile.
The attribute definition 0.FDO/Profile uses profile referencing. It contains the following two attributes:
  • (0.FDO/ValidationMechanism, ProfileReference): This means that the attribute definition 0.FDO/Profile applies the profile referencing mechanism.
  • (0.FDO/DataType, 0.FDO/ProfileDef): This means that if the attribute definition 0.FDO/Profile is instantiated, the value must be a reference to an FDO that instantiates 0.FDO/ProfileDef (i.e., it must be a reference to an FDO profile).

The Detailed Requirements for Profile Referencing are:

Format of an Attribute Definition that Applies Profile Referencing:
  • An attribute definition MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, ProfileReference).
  • The attribute definition MAY also specify a list of profile PIDs P1,...,Pn where at least one of them MUST validate (see ) the referenced FDO. This is done by adding the attribute (0.FDO/DataType, Pi), i=1,...,n, into the attribute definition. If 0.FDO/DataType is not contained in the attribute definition, it just means that the referenced entity is an FDO (hence, that the value is a validated reference complying to at least 0.FDO/Root).
Validation Rule:

To validate an FDO that contains an attribute applying profile referencing, it is checked that the value of this attribute is either a reference to another FDO following at least one of the profiles specified in the attribute definition, or an in-line serialization of an FDO that instantiates at least one of the specified profiles.

[H5-1f] Attribute Referencing
Attribute referencing is a generic mechanism for redirecting to one or multiple other attribute(s) in the same record. Attribute referencing can, for example, be used to group information for different purposes. Or it can be used very similarly to unions, i.e., to state that an attribute is effectively realized through one or several other attributes in the record, which follow more specific attribute definitions. To start with, here is an example:
The mandatory attribute 0.FDO/Data (and actually also the 0.FDO/Type attribute) apply attribute referencing. The reason for this is that data can appear in different forms (see also Figure 3):
  • Data may be given by a reference to an external bit-sequence. Then, the 0.FDO/Data attribute would have as its value a URL. (Figure 3, top-right)
  • Data could also be encapsulated in its own FDO, especially, if the FDO is a collection. Then, the0.FDO/Data attribute would have as its value a reference to an FDO, which possibly follows a specific profile (Figure 3, bottom-left).
  • Data could also be in-line in the record. Then, the 0.FDO/Data attribute would have as its value a simple data bit-sequence (Figure 3, top-left).
As 0.FDO/Data uses attribute referencing, all these different notions of data references can be unified in one common and standardized attribute definition 0.FDO/Data.

The detailed requirements for Attribute Referencing are:

Format of an Attribute Definition that Applies Attribute Referencing:

The following requirements must be fulfilled such that an attribute definition A applies attribute referencing:

  • The attribute definition A MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, AttributeReference).
  • The attribute definition A MAY specify a whitelist of attribute definition PIDs B1,...,Bn using the attribute (0.FDO/DataType, Bi), i=1,...,n, in its record. If A specifies this whitelist, an attribute a (instantiating A) MUST reference an attribute b that instantiates any of B1,...,Bn. If the 0.FDO/DataType attribute is not contained in A, any attribute b may be referenced by a, as there is no whitelist.
  • The cardinality of the attribute definition A only specifies the occurrence of instances of A itself (and not the occurrence of instances of any of B1,...,Bn). The cardinality of B1,...,Bn cannot be influenced by A.

Instantiating an attribute that applies attribute referencing

An attribute a=(adef,aval) (instantiating attribute definition A) is a reference to an attribute b=(bdef,bval) (instantiating attribute definition B) if and only if
  • aval is a PID resolving to an attribute definition
  • the referenced attribute b is instantiated in the same FDO record as attribute a
  • aval = bdef
  • bval is validated by bdef. This ensures, the reference is actually fulfilled.
See also Figures 4 and 5 for illustration. There may be cases of consecutive referencing (e.g., a references b, b references c, …). All such references within an FDO record must form a directed acyclic graph (no loops allowed).

Validation Rules:

To validate an FDO that contains an attribute a=(adef,aval) applying attribute referencing, it is checked that

  1. aval is a PID resolving to an attribute definition.
  2. There is another attribute b=(bdef,bval) in the FDO record such that aval = bdef.
  3. bval is validated by bdef. Note: This may involve checking validity of attributes c, d, ... if b is also an attribute applying attribute referencing (consecutive referencing).
If A is specified to apply attribute referencing but there is no attribute b that fulfills all validation conditions, then attribute a MAY be ignored to handle this error gracefully in certain use cases.
Attribute referencing is very similar to unions of attribute definitions: In both cases, an attribute definition states that its instantiation validates one of several attribute definitions. However, the slight difference is that for attribute referencing, the instantiating record also states which of the attribute definitions is actually chosen, while this is not stated in the case of unions.
[H5-1g] External Vocabulary Referencing
It is essential for FDOs that existing vocabularies, ontologies and terminologies can be reused in records (even if they are not in the strict format as described in [H5-1]). This can be carried out by applying external vocabulary referencing.

  • The attribute definition 0.FDO/SchemaOrgTerm states that it uses external attribute referencing through the attribute (0.FDO/ValidationMechanism, ExternalVocabularyReference).
  • The syntax definition 0.FDO/SchemaOrgTermSyntax specifies the syntax of the attribute definition, i.e., it must be instantiated by the string “https://schema.org”, which is the schema.org namespace.
  • The FDO 21.123/456 contains the attribute (0.FDO/SchemaOrgTerm, https://schema.org) to indicate that it uses vocabulary from schema.org. Afterwards, any schema.org term can be used as an attribute in the form (<URL of term>, <value>), such as (https://schema.org/priceCurrency, USD).

The detailed requirements for External Vocabulary Referencing are:

Format of an Attribute Definition that Applies External Vocabulary Referencing:

The following detailed requirements must be fulfilled such that an attribute definition A applies external vocabulary referencing:

  • The attribute definition A MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, ExternalVocabularyReference).
  • The attribute definition A MUST specify the namespace of the external vocabulary using the attribute 0.FDO/DataType which references a syntax definition that validates the namespace.

Instantiation of External Vocabulary Attributes in the Record:

To use external vocabulary in the FDO record, the record MUST contain
  • an attribute a=(adef,aval) that instantiates attribute definition A, where A applies external vocabulary referencing
  • an attribute b=(bdef,bval) representing a vocabulary term
The following requirements must be fulfilled by a and b:
  • aval is a URL representing a namespace
  • bdef = aval/<term>, i.e., bdef is a URL representing a term under a namespace
  • bval instantiates bdef.

Validation Rules:

To validate an FDO that contains an attribute a=(adef,aval) applying external vocabulary referencing, it is checked that the namespace aval is valid according to the syntax definition referenced from the attribute definition.
