Each section within this chapter explains in detail how the FDO building blocks from the technology-agnostic FDO specification are implemented by Handle-based FDOs.

PID & Resolution

[H1-1] PID System
Each Handle-based FDO has a Handle identifier as its PID, assigned through the Handle System.
[H1-2] PID Detection
The syntax of a Handle identifier follows the regular expression: `\d+(\.\d+)*\/\S+`
[H1-3] PID Resolution
The resolution of a Handle-based FDO, which is identified by PID X, into an FDO record uses the global Handle Resolver to receive the Handle record at `https://hdl.handle.net/api/handles/[X]`.

Attribute mapping

All mandatory and optional attributes that are given in the FDO spec are implemented as Handles. Handle-based FDOs make direct use of these attributes. Hence, no additional mappings are required.

FDO records

This section specifies how an FDO record is implemented by Handle-based FDOs.
Handle record
Each Handle record consists of a list of Handle values. Each Handle value is represented as a JSON object with (at least) five types (see also Handle Technical Manual):
  • index (integer)
  • type (string)
  • data (object)
  • ttl (the time-to-live in seconds of the value, an integer (or, in the rare case of an absolute expiration time, that expiration time as an ISO8601-formatted string))
  • timestamp (an ISO8601-formatted string)
FDO record
The conceptional FDO record (according to the FDO specification) consists of tuples `(k_1:v_1),(k_2:v_2),…,(k_n:v_n)`.
[H2-1] Mapping the FDO Record to the Handle Record
We assume an FDO record which consists of FDO attributes/tuples `a_1=(k_1:v_1),a_2=(k_2:v_2),…, a_n=(k_n:v_n)`. Mapping the FDO record to a Handle record means to map all FDO attributes `a_1,...,a_n` to a list of Handle values `h_1,...,h_n` as follows: Each attribute `a_i` is assigned a positive integer `f_i`, where `f_1, ..., f_n` are mutually different. Assigning integers to tuples should take conventions from Handle identifiers into account (e.g., if `k_i=HS_ADMIN`, then `f_i=100`). Then, the FDO attribute `a_i` is mapped to the Handle value `h_i` by setting:
  • `index: f_i`
  • `type: k_i`
  • `data.value: v_i`
The remaining types of Handle value `h_i` are usually set according to their default values, but possibly also with different values, if this makes sense in the given context.
  • `data.format: string` (and `data.format: admin`, if `k_i=HS_ADMIN`)
  • `ttl: 86400`
  • `timestamp: [current time]`
As an example, the FDO record with handle identifier `123/abcde` containing the three FDO attributes
  • `(0.FDO/Type: 123/45678)`
  • `(0.FDO/Profile: 123/00001)`
  • `(HS_ADMIN: {"handle": "0.NA/21.T11971", "index": 200, "permissions": "111111111111"})`
is mapped to the following Handle record in JSON format: 
{
    "responseCode": 1,
    "handle": "123/abcde",
    "values": [
        {
            "index": 1,
            "type": "0.FDO/Type",
            "data": {
                "format": "string",
                "value": "123/45678"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 2,
            "type": "0.FDO/Profile",
            "data": {
                "format": "string",
                "value": "123/00001"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 100,
            "type": "HS_ADMIN",
            "data": {
                "format": "admin",
                "value": {
                    "handle": "0.NA/21.T11971",
                    "index": 200,
                    "permissions": "111111111111"
                }
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        }
    ]
}

[H2-2] Mapping the Handle Record to the FDO Record
Mapping a handle record with handle values `h_1,...,h_n` to an FDO record with attributes `a_1,...,a_n` means to map each handle value `h_i` to an FDO attribute `a_i`. The handle value `h_i` 
{
    "index": f_i,
    "type": "k_i",
    "data": {
        "format": "string",
        "value": "v_i"
    },
    "ttl": 86400,
    "timestamp": "2025-05-06T12:30:44Z"
}
is mapped to the FDO attribute `a_i=(k_i, v_i)`. All other types (`index`, `data.value`, `ttl`, `timestamp`) are dropped.

FDO profiles

[H3-1] Representation of FDO Profiles
The Handle record of a Handle-based FDO profile must contain a fixed, predefined set of FDO attributes (see Tables 1 and 2). Note that there are two types of attributes in the Handle record:
  1. Meta information about the profile, to describe the FDO profile itself: `0.FDO/Type`, `0.FDO/Profile`, `0.FDO/Name` (all mandatory), `0.FDO/Description` (optional).
  2. The actual content of the profile, from which all validation rules (for FDOs instantiating this profile) can be derived. There are two ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute `0.FDO/Data`. For more details, see Table 1.
    • Way 2: The FDO profile is fully specified in the Handle record, simply relying on references to other attributes (`0.FDO/Attribute`), potential profile extension (`0.FDO/Extends`) and a boolean (`0.FDO/denyAdditionalAttributes`). For more details, see Table 2.
    • Note that both ways can be combined by using all attributes `0.FDO/Data`, `0.FDO/Attribute`, `0.FDO/Extends`, `0.FDO/denyAdditionalAttributes` simultaneously. In this case, when creating or updating this FDO profile, it must be checked that both ways induce the same validation rules.
Table 1: The list of attribute definitions which MUST be used when defining a new Handle-based FDO profile according to way 1. This table does not include all information about the attribute definitions. For more information, please resolve the attribute definition PIDs via the Handle System.
PID of the attribute definition Attribute value Cardinality of the attribute
0.FDO/Type The string FDO_Profile. Additional types may be added, if applicable. 1..*
0.FDO/Profile The string 0.FDO/ProfileDef. Additional profiles may be added, if applicable. 1..*
0.FDO/Name Human-readable and localized label(s) for the profile. 1..*
0.FDO/Description Human-readable and localized description(s) for the profile. 0..*
0.FDO/Data URL pointing to a JSON schema which represents an FDO profile. The JSON schema must have a list of FDO attribute PIDs as its properties, which contains at least the mandatory attributes 0.FDO/Type, 0.FDO/Profile, 0.FDO/Data. Furthermore, it may specify that it extends another FDO profile by using the `$ref` keyword, referencing the JSON schema of a parent profile. It may also specify whether it is allowed for instantiating FDOs to contain additional attributes by using the `additionalProperties` keyword. 1

Table 2: The list of attribute definitions which MUST be used when defining a new Handle-based FDO profile according to way 2. This table does not include all information about the attribute definitions. For more information, please resolve the attribute definition PIDs via the Handle System.
PID of the attribute definition Attribute value Cardinality of the attribute
0.FDO/Type The string FDO_Profile. Additional types may be added, if applicable. 1..*
0.FDO/Profile The string 0.FDO/ProfileDef. Additional profiles may be added, if applicable. 1..*
0.FDO/Name Human-readable and localized label(s) for the profile. 1..*
0.FDO/Description Human-readable and localized description(s) for the profile. 0..*

Additional attributes can be added by further repeating the 0.FDO/Attribute with other values. 3..*
0.FDO/Extends If this profile is an extension of another profile, this attribute specifies the PID of the parent profile (see [H3-3]). 0..*
0.FDO/denyAdditionalAttributes This is an optional boolean flag indicating whether an FDO that instantiates the respective profile MUST NOT have additional attributes than those given by the profile. This restriction is only active if the value is set to a truthy boolean. 0..1

[H3-3] FDO Profiles Extension Mechanism
  • An FDO profile constructed as JSON Schema using way 1 references a parent JSON schema from a parent FDO profile using the keyword `$ref`.
  • An FDO profile constructed using way 2 references a parent FDO profile using the attribute `0.FDO/Extends`. Its value must be a PID pointing to another FDO profile.
