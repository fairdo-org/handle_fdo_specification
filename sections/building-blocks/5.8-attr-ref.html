[H5-8] Attribute Referencing
Attribute referencing is a generic mechanism for redirecting to one or multiple other attribute(s) in the same record. Attribute referencing can, for example, be used to group information for different purposes. Or it can be used very similarly to unions, i.e., to state that an attribute is effectively realized through one or several other attributes in the record, which follow more specific attribute definitions. To start with, here is an example for FDO records that instantiate attributes that apply attribute referencing:
The mandatory attribute 21.T11969/Data (and actually also the 21.T11969/Type attribute) applies attribute referencing. The reason for this is that data can appear in different forms (see also Figure 3):
  • Data could also be in-line in the record. Then, the 21.T11969/Data attribute would have as its value a simple data bit-sequence (a).
  • Data may be given by a reference to an external bit-sequence. Then, the 21.T11969/Data attribute would have as its value a URL (b).
  • Data could also be encapsulated in its own FDO, especially, if the FDO is a collection. Then, the 21.T11969/Data attribute would have as its value a reference to an FDO, which possibly follows a specific profile (c).
As 21.T11969/Data uses attribute referencing, all these different notions of data references can be unified in one common and standardized attribute definition 21.T11969/Data.
Attribute referencing is very similar to unions of attribute definitions: In both cases, an attribute definition states that its instantiation validates one of several attribute definitions. However, there are two slight differences in comparison to unions:
  1. For attribute referencing, one can also reference to an attribute that does not have an attribute definition (i.e., which is simply a human-readable label without any validation rules).
  2. An attribute applying attribute referencing explicitly states which attribute definition the referenced value complies to, which is not the case for unions.
Instantiating an attribute that applies attribute referencing: An attribute a1 = (k1: v1) (instantiating attribute definition A1) is a reference to an attribute a2 = (k2: v2) if and only if
  • the referenced attribute a2 is instantiated in the same FDO record as attribute a1 such that v1 = k2
  • If k2 is an attribute PID (i.e., a2 instantiates an attribute definition A2), then v2 must be valid with respect to k2. This ensures, the reference is actually fulfilled.
See also for an illustration. There may be cases of consecutive referencing (e.g., a1 references a2, a2 references a3, …). All such references within an FDO record must form a directed acyclic graph (i.e., no loops allowed).

Validation Rules: To validate an FDO that contains an attribute a = (k1: v1) applying attribute referencing, it is checked that

  1. There is another attribute a2 = (k2: v2) in the same FDO record such that v1 = k2.
  2. If k2 is an attribute PID, v2 must be validated by k2. Note: This may involve checking validity of attributes a3, a4, ... if a2 is also an attribute applying attribute referencing (consecutive referencing).
If attribute a1 is an attribute reference but there is no referenced attribute a2 in the record, then a1 MAY be ignored to handle this error gracefully in certain use cases.
There are two (almost) equivalent ways how to build a Handle-based attribute definition that applies attribute referencing (see [H5-8a] and [H5-8b]).
[H5-8a] allows to reference to any attribute, while [H5-8b] only allows to reference to an attribute that instantiates an attribute definition. Hence, [H5-8a] is a bit more expressive.

[H5-8a] Attribute Referencing implemented by JSON Schemas:

The FDO record of the attribute definition A must contain a JSON schema which encodes an attribute reference to one of the attributes b1,...,bn. In detail, the attribute 21.T11969/Data applies attribute referencing to point to the attribute 21.T11969/JsonSchema, which has as its value a JSON schema. The JSON schema can specify both syntax rules for the attribute value (i.e. that it is a whitelist of strings allowing for b1,...,bn) and that it applies attribute referencing. The validity of attribute referencing must be checked on top of simple JSON schema validation. 
{
    "enum": ["[PID or name of b1]", "...", "[PID or name of bn]"],
    "validation": {
        "mechanism": "attributeReference"
    }
}
If the enum is left out, it means that the referenced attribute can be anything, e.g. an attribute instantiating an attribute definition, or a simple label.
The attribute definition 21.T11969/Data uses attribute referencing. An implementation of its FDO record could look like this:
PID of Attribute Definition Example Value
21.T11969/Type FDO_Attribute
21.T11969/Profile 21.T11969/AttributeDef
21.T11969/Name Data
21.T11969/Description Reference to the data bit-sequence of the FDO.
21.T11969/Cardinality 1..n
21.T11969/Data 21.T11969/JsonSchema
21.T11969/JsonSchema 
{
    "$schema": "http://json-schema.org/draft-04/schema#",
    "@id": "hdl:21.T11969/Data",
    "title": "Data",
    "description": "Reference to the data bit-sequence of the FDO.",
    "type": "string",
    "enum": ["21.T11969/Attr_Def_inline_data", "21.T11969/Attr_Def_URL", "21.T11969/Attr_Def_FDO_Ref"],
    "validation": {
        "mechanism": "attributeReference"
    }
}
FDO record of attribute definition (PID: 21.T11969/Data.)

[H5-8b] Attribute Referencing implemented on Handle-record level:

The FDO record of the attribute definition A must specify the attributes 21.T11969/ValidationMechanism and 21.T11969/DataType to define an attribute reference to one of the attribute definitions B1,...,Bn. In detail, the following attributes MUST be in the attribute definition:
  • (21.T11969/ValidationMechanism, AttributeReference) to state that this attribute definition uses attribute referencing
  • (21.T11969/DataType, B1), ..., (21.T11969/DataType, Bn) to state that the attribute value must point to an attribute in the same record that instantiates one of the attribute definitions B1, ..., Bn. If 21.T11969/DataType is not contained in the attribute definition, it means that the referenced attribute can be anything, e.g. an attribute following any attribute definition, or even a simple label.
  • The cardinality of the attribute definition A only specifies the occurrence of instances of A itself (and not the occurrence of instances of any of B1,...,Bn). The cardinality of B1,...,Bn cannot be influenced by A.
The attribute definition 21.T11969/Data uses attribute referencing. Its FDO record could look like this:
