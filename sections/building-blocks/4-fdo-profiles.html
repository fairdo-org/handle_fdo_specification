FDO profiles

[H4-1] Representation of FDO Profiles
The Handle record of a Handle-based FDO profile must contain a fixed, predefined set of FDO attributes (see Table 3). Note that there are two types of attributes in the Handle record:
  1. Meta information about the profile, to describe the FDO profile itself: 21.T11969/Type, 21.T11969/Profile, 21.T11969/Name (all mandatory), 21.T11969/Description (optional).
  2. The actual content of the profile, from which validation rules (for FDOs instantiating this profile) can be derived: 21.T11969/Data, 21.T11969/JsonSchema, 21.T11969/Attribute, 21.T11969/Extends, and 21.T11969/denyAdditionalAttributes. There are two equivalent ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains validation rules. The linking is done by using the attribute 21.T11969/Data. This attribute applies attribute referencing to point to the attribute 21.T11969/JsonSchema, which has as its value a URL. This URL references a JSON schema. The JSON schema must have a list of FDO attribute PIDs as its properties, which contains at least the mandatory attributes 21.T11969/Type, 21.T11969/Profile, and 21.T11969/Data. Furthermore, it may specify that it extends another FDO profile by using the $ref keyword to reference the JSON schema of a parent profile. It may also specify whether it allows instantiating FDOs to contain additional attributes by using the additionalProperties keyword.

    • Way 2: The FDO profile is fully specified in the Handle record, simply relying on references to other attributes (`21.T11969/Attribute`), potential profile extension (`21.T11969/Extends`) and a boolean (`21.T11969/denyAdditionalAttributes`). In more detail:
      • The attrinbute 21.T11969/Attribute defines the list of attribute definitions that MUST be instantiated in any FDO instantiating this profile. In case that no extension is used, the attribute 0.FDO/Attribute MUST be repeated at least three times, once with each of the values 0.FDO/Type, 0.FDO/Profile, 0.FDO/Data, because these are the mandatory attributes that MUST be contained in any FDO. Additional attributes can be added by further repeating the 0.FDO/Attribute with other values.
      • If this profile is an extension of another profile, the attribute 21.T11969/Extends specifies the PID of the parent profile (see [H4-3]).
      • The attribute 21.T11969/denyAdditionalAttributes is an optional boolean flag indicating whether an FDO that instantiates the respective profile MUST NOT have additional attributes than those given by the profile. This restriction is only active if the value is set to a truthy boolean.

    • Note that both ways can be combined by using all attributes 21.T11969/Data, 21.T11969/JsonSchema, 21.T11969/Attribute, 21.T11969/Extends, and 21.T11969/denyAdditionalAttributes simultaneously. In this case, when creating or updating this FDO profile, it must be checked that both ways induce the same validation rules.

[H4-3] FDO Profiles Extension Mechanism
  • An FDO profile constructed as JSON Schema using way 1 references a parent JSON schema from a parent FDO profile using the keyword `$ref`.
  • An FDO profile constructed using way 2 references a parent FDO profile using the attribute `0.FDO/Extends`. Its value must be a PID pointing to another FDO profile.
