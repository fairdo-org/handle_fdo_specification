FDO profiles

[H4-1] Representation of FDO Profiles
The Handle record of a Handle-based FDO profile must contain a fixed, predefined set of FDO attributes (see Table 3). Note that there are two types of FDO attributes in the Handle record:
  1. Meta information about the profile, to describe the FDO profile itself: 21.T11969/Type, 21.T11969/Profile, 21.T11969/Name (all mandatory), 21.T11969/Description (optional).
  2. The actual content of the profile, from which validation rules (for FDOs instantiating this profile) can be derived: 21.T11969/Data, 21.T11969/JsonSchema, 21.T11969/Attribute, 21.T11969/Extends, and 21.T11969/denyAdditionalAttributes. There are two equivalent ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains validation rules.
      • The linking is done by using the attribute 21.T11969/Data. This attribute applies attribute referencing to point to the attribute 21.T11969/JsonSchema, which has as its value a URL. This URL references a JSON schema.
      • The JSON schema must have attribute PIDs as its properties. If the profile is not an extension of an existing profile, it must at least contain the mandatory attributes 21.T11969/Type, 21.T11969/Profile, and 21.T11969/Data. Furthermore, it may specify that it extends another FDO profile by using the JSON schema $ref keyword to reference the JSON schema of a parent profile. It may also specify whether it allows instantiating FDOs to contain additional attribute by using the additionalProperties keyword.
      The Handle record of a profile that is constructed according to way 1 looks as follows:
      PID of Attribute Definition Example Value
      21.T11969/Type FDO_Profile
      21.T11969/Profile 21.T11969/ProfileDef
      21.T11969/Name Simple_Profile
      21.T11969/Description This is a very simple profile which only contains the mandatory attributes and a reference to a metadata file.
      21.T11969/Data 21.T11969/JsonSchema
      21.T11969/JsonSchema https://typeapi.lab.pidconsortium.net/v1/types/schema/21.example/profile01
      Profile FDO record (fictionary PID: 21.example/profile01)
      The fictionary URL https://typeapi.lab.pidconsortium.net/v1/types/schema/21.example/profile01, the value of the attribute 21.T11969/JsonSchema, references a fictionary JSON schema. The JSON schema could look like this: 
      {
    "$schema": "http://json-schema.org/draft-04/schema#",
    "title": "Simple_Profile",
    "description": "This is a very simple profile which only contains the mandatory attributes and a reference to a
                                metadata file.",
    "type": "object",
    "properties": {
        "21.T11969/Type": {
            "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/Type"
        },
        "21.T11969/Profile": {
            "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/Profile"
        },
        "21.T11969/Data": {
            "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/Data"
        },
        "21.T11969/Metadata": {
            "title": "Metadata_file",
            "description": "URL pointing to a file that contains metadata",
            "type": "string",
            "pattern": "^http(s)?:(\/\/([^/?#]*))?([^?#]*)(\?([^#]*))?(#(.*))?$"
        }
    },
    "additionalProperties": false
}
    • Way 2: The FDO profile is fully specified in the Handle record, simply relying on references to other attributes, potential profile extension and a boolean. In more detail:
      • The attribute 21.T11969/Attribute defines the list of attribute definitions that MUST be instantiated in any FDO instantiating this profile. In case that no extension is used, the attribute 21.T11969/Attribute MUST be repeated at least three times, once with each of the values 21.T11969/Type, 21.T11969/Profile, 21.T11969/Data, because these are the mandatory attribute that MUST be contained in any FDO. Additional attribute can be added by further repeating the 21.T11969/Attribute with other values.
      • If this profile is an extension of another profile, the value of the attribute 21.T11969/Extends specifies the PID of the parent profile (see [H4-3]).
      • The attribute 21.T11969/denyAdditionalAttributes is an optional boolean flag indicating whether an FDO that instantiates the respective profile MUST NOT have additional attribute than those given by the profile. This restriction is only active if the value is set to a truthy boolean.
      • The mandatory attribute 21.T11969/Data is set to Not_Applicable.
      The same profile as in looks like this when it is represented using way 2:
      PID of Attribute Definition Example Value
      21.T11969/Type FDO_Profile
      21.T11969/Profile 21.T11969/ProfileDef
      21.T11969/Name Simple_Profile
      21.T11969/Description This is a very simple profile which only contains the mandatory attributes and a reference to a metadata file.
      21.T11969/Data Not_Applicable
      21.T11969/Attribute 21.T11969/Type
      21.T11969/Attribute 21.T11969/Profile
      21.T11969/Attribute 21.T11969/Data
      21.T11969/Attribute 21.T11969/Metadata
      21.T11969/denyAdditionalAttributes True
      Profile FDO record (fictionary PID: 21.example/profile01)
    • Both ways can be combined by using all attributes 21.T11969/Data, 21.T11969/JsonSchema, 21.T11969/Attribute, 21.T11969/Extends, and 21.T11969/denyAdditionalAttributes simultaneously. In this case, when creating or updating this FDO profile, it must be checked that both ways induce the same validation rules.
Table 3: The list of attribute definitions which MUST be used when defining a new Handle-based FDO profile according to way 2. This table does not include all information about the attribute definitions. For more information, please resolve the attribute definition PIDs via the Handle System.
PID of the attribute definition Attribute value Cardinality of the attribute
21.T11969/Type The string FDO_Profile. Additional types may be added, if applicable. 1..*
21.T11969/Profile The string 21.T11969/ProfileDef. Additional profiles may be added, if applicable. 1..*
21.T11969/Name Human-readable and localized label(s) for the profile. 1..*
21.T11969/Description Human-readable and localized description(s) for the profile. 0..*
21.T11969/Data Either the string 21.T11969/JsonSchema (way 1) or the string Not_Applicable (way 2). If both ways are combined, it has the value of way 1. 1
21.T11969/JsonSchema This attribute is just used when the attribute is represented according to way 1. In this case, it is a URL pointing to a JSON schema. 0..1
21.T11969/Attribute This attribute is used when the attribute is represented according to way 2. The attribute 21.T11969/Attribute defines the list of attribute definitions that MUST be instantiated in any FDO using this profile. The attribute 21.T11969/Attribute is repeated once for each attribute definition that should be in the profile. 0..*
21.T11969/Extends This attribute is just used when the attribute is represented according to way 2. If this profile is an extension of another profile, the value of this attribute specifies the PID of the parent profile. 0..*
21.T11969/denyAdditionalAttributes This attribute is just used when the attribute is represented according to way 2. The value is an optional boolean flag indicating whether an FDO that instantiates the respective profile MUST NOT have additional attributes than those given by the profile. This restriction is only active if the value is set to a truthy boolean. 0..1

[H4-2] Profile Definition Profile
The Handle-based profile definition profile has the PID 21.T11969/ProfileDef and is the Handle-based implementation of Table 3. It uses both mechanisms way 1 and way 2 (see [H4-1]).
[H4-3] Root Profile
The Handle-based root profile has the PID 21.T11969/Root. It uses both mechanisms way 1 and way 2 (see [H4-1]). It can be used as a parent profile for profile extension.
[H4-4] FDO Profiles Extension Mechanism
  • An FDO profile constructed as JSON Schema (way 1) embeds a parent JSON schema from a parent FDO profile through a combination of `allOf` and `$ref`.
    When you use JSON schemas in combination with profile extension, it is technically not possible to deny additional attributes. If you need to use extension and deny additional attributes, you have to go for way 2 to represent your profile.
    We represent the profile from using way 1 and profile extension: The Hande record is exactly the same as in . The JSON schema looks like this: 
    {
    "$schema": "http://json-schema.org/draft-04/schema#",
    "title": "Simple_Profile",
    "description": "This is a very simple profile which only contains the mandatory attributes and a reference to a
                                metadata file.",
    "type": "object",
    "allOf": [{ "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/Root"}],
    "properties": {
        "21.T11969/Metadata": {
            "title": "Metadata_file",
            "description": "URL pointing to a file that contains metadata",
            "type": "string",
            "pattern": "^http(s)?:(\/\/([^/?#]*))?([^?#]*)(\?([^#]*))?(#(.*))?$"
        }
    }
}
    However, it was not possible to keep the additionalProperties: false.
  • An FDO profile constructed on Handle record level (way 2) references a parent FDO profile using the attribute 21.T11969/Extends. Its value must be a PID pointing to another FDO profile.
    The same profile as in looks like this when it is represented using way 2 in combination with profile extension:
    PID of Attribute Definition Example Value
    21.T11969/Type FDO_Profile
    21.T11969/Profile 21.T11969/ProfileDef
    21.T11969/Name Simple_Profile
    21.T11969/Description This is a very simple profile which only contains the mandatory attributes and a reference to a metadata file.
    21.T11969/Data Not_Applicable
    21.T11969/Attribute 21.T11969/Metadata
    21.T11969/Extends 21.T11969/Root
    21.T11969/denyAdditionalAttributes True
    Profile FDO record (fictionary PID: 21.example/profile01)
