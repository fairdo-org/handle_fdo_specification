FDO profiles

[H4-1] Representation of FDO Profiles
The Handle record of a Handle-based FDO profile must contain a fixed, predefined set of FDO attributes (see Tables 1 and 2). Note that there are two types of attributes in the Handle record:
  1. Meta information about the profile, to describe the FDO profile itself: `0.FDO/Type`, `0.FDO/Profile`, `0.FDO/Name` (all mandatory), `0.FDO/Description` (optional).
  2. The actual content of the profile, from which all validation rules (for FDOs instantiating this profile) can be derived. There are two ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute `0.FDO/Data`. For more details, see Table 1.
    • Way 2: The FDO profile is fully specified in the Handle record, simply relying on references to other attributes (`0.FDO/Attribute`), potential profile extension (`0.FDO/Extends`) and a boolean (`0.FDO/denyAdditionalAttributes`). For more details, see Table 2.
    • Note that both ways can be combined by using all attributes `0.FDO/Data`, `0.FDO/Attribute`, `0.FDO/Extends`, `0.FDO/denyAdditionalAttributes` simultaneously. In this case, when creating or updating this FDO profile, it must be checked that both ways induce the same validation rules.
Table 1: The list of attribute definitions which MUST be used when defining a new Handle-based FDO profile according to way 1. This table does not include all information about the attribute definitions. For more information, please resolve the attribute definition PIDs via the Handle System.
PID of the attribute definition Attribute value Cardinality of the attribute
0.FDO/Type The string FDO_Profile. Additional types may be added, if applicable. 1..*
0.FDO/Profile The string 0.FDO/ProfileDef. Additional profiles may be added, if applicable. 1..*
0.FDO/Name Human-readable and localized label(s) for the profile. 1..*
0.FDO/Description Human-readable and localized description(s) for the profile. 0..*
0.FDO/Data URL pointing to a JSON schema which represents an FDO profile. The JSON schema must have a list of FDO attribute PIDs as its properties, which contains at least the mandatory attributes 0.FDO/Type, 0.FDO/Profile, 0.FDO/Data. Furthermore, it may specify that it extends another FDO profile by using the `$ref` keyword, referencing the JSON schema of a parent profile. It may also specify whether it is allowed for instantiating FDOs to contain additional attributes by using the `additionalProperties` keyword. 1

Table 2: The list of attribute definitions which MUST be used when defining a new Handle-based FDO profile according to way 2. This table does not include all information about the attribute definitions. For more information, please resolve the attribute definition PIDs via the Handle System.
PID of the attribute definition Attribute value Cardinality of the attribute
0.FDO/Type The string FDO_Profile. Additional types may be added, if applicable. 1..*

[H4-3] FDO Profiles Extension Mechanism
  • An FDO profile constructed as JSON Schema using way 1 references a parent JSON schema from a parent FDO profile using the keyword `$ref`.
  • An FDO profile constructed using way 2 references a parent FDO profile using the attribute `0.FDO/Extends`. Its value must be a PID pointing to another FDO profile.
