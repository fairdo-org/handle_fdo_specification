FDO records

This section specifies how an FDO record is implemented by Handle-based FDOs.
Handle record
Each Handle record consists of a list of Handle values. Each Handle value is represented as a JSON object with (at least) five types (see also [[HandleTechnical]]):
  • index (integer)
  • type (string)
  • data (object)
  • ttl (the time-to-live in seconds of the value, an integer (or, in the rare case of an absolute expiration time, that expiration time as an ISO8601-formatted string))
  • timestamp (an ISO8601-formatted string)
FDO record
The conceptional FDO record (according to the FDO specification) consists of tuples (k1: v1), (k2: v2),..., (kn: vn), where each ki is either a PID identifying an attribute definition, or a human-readable label of an attribute.
[H3-1] Mapping the FDO Record to the Handle Record

We assume an FDO record which consists of FDO attributes/tuples a1 = (k1 : v1), a2 = (k2 : v2),..., an = (kn : vn). Mapping the FDO record to a Handle record means to map all FDO attributes a1,...,an to a list of Handle values h1,...,hn as follows:

Each attribute ai is assigned a positive integer fi, where f1, ..., fn are mutually different. Assigning integers to tuples should take conventions from Handle identifiers into account (e.g., if ki = HS_ADMIN, then fi = 100). Then, the FDO attribute ai is mapped to the Handle value hi by setting:

  • index: fi
  • type: ki' (Here, ki' = ki if ki is an attribute name or a community attribute, and ki' = 21.T11969/[X] if ki is a conceptual attribute PID 0.FDO/[X] that is listed in Table 1 or Table 2.)
  • data.value: vi
The remaining types of Handle value hi are usually set according to their default values, but possibly also with different values, if this makes sense in the given context:
  • data.format: string (and data.format: admin, if ki = HS_ADMIN)
  • ttl: 86400
  • timestamp: [current time]
The FDO record with handle identifier 123/abcde containing the four FDO attributes
  • (0.FDO/Type: 123/45678)
  • (0.FDO/Profile: 123/00001)
  • (version: 1.0)
  • (HS_ADMIN: {"handle": "0.NA/21.T11971", "index": 200, "permissions": "111111111111"})
is mapped to the following Handle record in JSON format: 
{
    "responseCode": 1,
    "handle": "123/abcde",
    "values": [
        {
            "index": 1,
            "type": "21.T11969/Type",
            "data": {
                "format": "string",
                "value": "123/45678"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 2,
            "type": "21.T11969/Profile",
            "data": {
                "format": "string",
                "value": "123/00001"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 3,
            "type": "version",
            "data": {
                "format": "string",
                "value": "1.0"
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        },
        {
            "index": 100,
            "type": "HS_ADMIN",
            "data": {
                "format": "admin",
                "value": {
                    "handle": "0.NA/21.T11971",
                    "index": 200,
                    "permissions": "111111111111"
                }
            },
            "ttl": 86400,
            "timestamp": "2025-05-06T12:30:44Z"
        }
    ]
}
[H3-2] Mapping the Handle Record to the FDO Record
Mapping a handle record with handle values h1,...,hn to an FDO record with attributes a1,...,an means to map each handle value hi to an FDO attribute ai. The handle value hi with
  • index: fi
  • type: ki
  • data.format: string
  • data.value: vi
  • ttl: 86400
  • timestamp: 2025-05-06T12:30:44Z
is mapped to the FDO attribute ai = (ki', vi), where ki' = ki if ki is an attribute name or a handle-based community attribute, and ki' = 0.FDO/[X] if ki is a Handle-based attribute PID 21.T11969/[X] that is listed in Table 1 or Table 2. All other types (index, data.value, ttl, timestamp) are dropped. This exactly reverses the mapping in [H3-1].
