FDO attribute definitions

[H5-1] Representation of FDO attribute definitions
The Handle record of a Handle-based FDO attribute definition must contain a fixed, predefined set of FDO attributes (see Tables 3 and 4). Note that there are two types of attributes in the Handle record:
  1. Meta information about the attribute definition, to describe the FDO attribute definition itself: 21.T11969/Type, 21.T11969/Profile, 21.T11969/Name, 21.T11969/Cardinality (all mandatory), 21.T11969/Description (optional).
  2. The actual content of the attribute definition, from which all validation rules (for attributes instantiating this attribute definition) can be derived. There are two ways how to define the validation rules:
    • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute 21.T11969/Data. For more details, see Table 3.
    • Way 2: The FDO attribute definition is fully specified in the Handle record, simply relying on references to other attribute definitions, syntax definitions, or profiles (21.T11969/DataType) and a specified validation mechanism (`21.T11969/ValidationMechanism`). For more details, see Table 4.
    • Note that both ways can be combined by using all attributes 21.T11969/Data, 21.T11969/DataType, 21.T11969/ValidationMechanism simultaneously. In this case, when creating or updating this FDO attribute definition, it must be checked that both ways induce the same validation rules.

[H5-1a] Encoding Semantics
A name must be and a description may be included in a Handle-based attribute definition using the attributes `21.T11969/Name` respectively `21.T11969/Description` in the FDO record. If way 1 is used, the name and the description can be repeated in the JSON schema using the keywords `title` and `description`.
[H5-1b] Syntax Definitions
  • Way 1: The FDO record links to a JSON schema which contains all validation rules using the attribute 21.T11969/Data. The JSON Schema encodes the following basic validation rules:
    1. An attribute states to follow one of the primitive data types “string”, “number”, “integer” and “boolean” using the `type` keyword.
    2. An attribute states to comply with a regular expression using the `pattern` keyword.
    3. An attribute states to lie within a numeric range using the keywords `minimum`, `exclusiveMinimum`, `maximum` or `exclusiveMaximum`.
    4. An attribute states to be equal to one element from a whitelist through the `enum` keyword.
    5. An attribute states to be not equal to all elements from a blacklist through combining `not` with `enum`. 
    {
"$schema": "http://json-schema.org/draft-04/schema#",
"title": "publication year",
"description": "The publication year of the resource.",
"type": "integer",
"minimum": 0,
"maximum": 2026
}
    {
"$schema": "http://json-schema.org/draft-04/schema#",
"title": "color",
"description": "The color of the resource.",
"type": "string",
"enum": ["red", "yellow", "green", "blue"],
"not":
{
    "enum": ["black", "white"]
}
}
  • Way 2: To specify the syntax of an attribute, the following two attributes must be used in the FDO record:
    1. (21.T11969/ValidationMechanism, Syntax)
    2. (21.T11969/Data, [reference to a syntax definition]).
    Syntax definitions are themselves FDOs and encode the basic validation rules in their records. They follow the syntax definition profile with the PID 21.T11969/SyntaxDef and contain the attributes as in table 5.

[H5-1c] Complex attributes
  • Way 1:
    1. A composition of attributes is implemented in JSON schema as an object with properties. Each property links to the JSON schema of another attribute definitions using the keyword `$ref`. You can use the `required` keyword to make attributes within the composition mandatory and `additionalProperties` to prevent adding additional content to the composition. 
      {
"$schema": "http://json-schema.org/draft-04/schema#",
"title": "author",
"description": "An author is a composition of a first name and a last name.",
"type": "object",
"properties": {
"firstName": {
  "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/3df63b7acb0522da685d"
},
"lastName": {
  "$ref": "https://typeapi.lab.pidconsortium.net/v1/types/schema/21.T11969/3df63b7acb0522da685d"
}
},
"required": ["firstName", "lastName"],
"additionalProperties": false
}
      This attribute composition validates: 
      {
 "firstName": "Jon",
 "lastName": "Doe"
}

[H5-1e] Profile Referencing
Profile referencing permits an attribute definition A to specify that the value aval (of an attribute a=(adef,aval) that instantiates A) MUST be a reference to an FDO following a specific profile. Hence, it is a mechanism to enforce that specific attributes in FDO record reference other FDOs that follow a certain profile.
The attribute definition 21.T11969/Profile uses profile referencing. It contains the following two attributes:
  • (21.T11969/ValidationMechanism, ProfileReference)): This means that the attribute definition 21.T11969/Profile applies the profile referencing mechanism.
  • (21.T11969/Data, 21.T11969/ProfileDef): This means that if the attribute definition 21.T11969/Profile is instantiated, the value must be a reference to an FDO that instantiates 21.T11969/ProfileDef (i.e., it must be a reference to an FDO profile).

The Detailed Requirements for Profile Referencing are:

Format of an attribute definition that Applies Profile Referencing:
  • An attribute definition MUST explicitly state that it uses this feature by containing the attribute (21.T11969/ValidationMechanism, ProfileReference).
  • The attribute definition MAY also specify a list of profile PIDs P1,...,Pn where at least one of them MUST validate the referenced FDO. This is done by adding the attribute (21.T11969/Data, Pi), i=1,...,n, into the attribute definition. If 21.T11969/Data is not contained in the attribute definition, it just means that the referenced entity is an FDO (hence, that the value is a validated reference complying to at least 21.T11969/Root).
Validation Rule:

To validate an FDO that contains an attribute applying profile referencing, it is checked that the value of this attribute is either a reference to another FDO following at least one of the profiles specified in the attribute definition, or an in-line serialization of an FDO that instantiates at least one of the specified profiles.

[H5-1f] Attribute Referencing
Attribute referencing is a generic mechanism for redirecting to one or multiple other attribute(s) in the same record. Attribute referencing can, for example, be used to group information for different purposes. Or it can be used very similarly to unions, i.e., to state that an attribute is effectively realized through one or several other attributes in the record, which follow more specific attribute definitions. To start with, here is an example:
The mandatory attribute 21.T11969/Data (and actually also the 21.T11969/Type attribute) apply attribute referencing. The reason for this is that data can appear in different forms (see also Figure 3):
  • Data may be given by a reference to an external bit-sequence. Then, the 21.T11969/Data attribute would have as its value a URL. (Figure 3, top-right)
  • Data could also be encapsulated in its own FDO, especially, if the FDO is a collection. Then, the 21.T11969/Data attribute would have as its value a reference to an FDO, which possibly follows a specific profile (Figure 3, bottom-left).
  • Data could also be in-line in the record. Then, the 21.T11969/Data attribute would have as its value a simple data bit-sequence (Figure 3, top-left).
As 21.T11969/Data uses attribute referencing, all these different notions of data references can be unified in one common and standardized attribute definition 21.T11969/Data.

The detailed requirements for Attribute Referencing are:

Format of an attribute definition that applies Attribute Referencing:

The following requirements must be fulfilled such that an attribute definition A applies attribute referencing:

  • The attribute definition A MUST explicitly state that it uses this feature by containing the attribute (21.T11969/ValidationMechanism, AttributeReference).
  • The attribute definition A MAY specify a whitelist of attribute definition PIDs B1,...,Bn using the attribute (21.T11969/Data, Bi), i=1,...,n, in its record. If A specifies this whitelist, an attribute a (instantiating A) MUST reference an attribute b that instantiates any of B1,...,Bn. If the 21.T11969/Data attribute is not contained in A, any attribute b may be referenced by a, as there is no whitelist.
  • The cardinality of the attribute definition A only specifies the occurrence of instances of A itself (and not the occurrence of instances of any of B1,...,Bn). The cardinality of B1,...,Bn cannot be influenced by A.

Instantiating an attribute that applies attribute referencing

An attribute a=(adef,aval) (instantiating attribute definition A) is a reference to an attribute b=(bdef,bval) (instantiating attribute definition B) if and only if
  • avalis a PID resolving to an attribute definition
  • the referenced attribute b is instantiated in the same FDO record as attribute a
  • aval = bdef
  • bval is validated by bdef. This ensures, the reference is actually fulfilled.
See also Figures 4 and 5 for illustration. There may be cases of consecutive referencing (e.g., a references b, b references c, …). All such references within an FDO record must form a directed acyclic graph (i.e., no loops allowed).

Validation Rules:

To validate an FDO that contains an attribute a=(adef,aval) applying attribute referencing, it is checked that

  1. >aval is a PID resolving to an attribute definition.
  2. There is another attribute b=(bdef,bval) in the FDO record such that >aval = bdef.
  3. bval is validated by bdef. Note: This may involve checking validity of attributes c, d, ... if b is also an attribute applying attribute referencing (consecutive referencing).
If A is specified to apply attribute referencing but there is no attribute b that fulfills all validation conditions, then attribute a MAY be ignored to handle this error gracefully in certain use cases.
Attribute referencing is very similar to unions of attribute definitions: In both cases, an attribute definition states that its instantiation validates one of several attribute definitions. However, the slight difference is that for attribute referencing, the instantiating record also states which of the attribute definitions is actually chosen, while this is not stated in the case of unions.
[H5-1g] External Vocabulary Referencing
It is essential for FDOs that existing vocabularies, ontologies, and terminologies can be reused in records (even if they are not in the strict format as described in [H5-1]). This can be carried out by applying external vocabulary referencing.
Figure 6 shows how schema.org terms can be used in an FDO record by applying external vocabulary referencing.
  • The attribute definition 21.T11969/SchemaOrgTerm states that it uses external attribute referencing through the attribute (21.T11969/ValidationMechanism, ExternalVocabularyReference).
  • The syntax definition 21.T11969/SchemaOrgTermSyntax specifies the syntax of the attribute definition, i.e., it must be instantiated by the string “https://schema.org”, which is the schema.org namespace.
  • The FDO 21.123/456 contains the attribute (21.T11969/SchemaOrgTerm, https://schema.org) to indicate that it uses vocabulary from schema.org. Afterwards, any schema.org term can be used as an attribute in the form (<URL of term>, <value>), such as (https://schema.org/priceCurrency, USD).

The detailed requirements for External Vocabulary Referencing are:

Format of an attribute definition that Applies External Vocabulary Referencing:

The following detailed requirements must be fulfilled such that an attribute definition A applies external vocabulary referencing:

  • The attribute definition A MUST explicitly state that it uses this feature by containing the attribute (21.T11969/ValidationMechanism, ExternalVocabularyReference).
  • The attribute definition A MUST specify the namespace of the external vocabulary using the attribute 21.T11969/Data which references a syntax definition that validates the namespace.

Instantiation of External Vocabulary Attributes in the Record:

To use external vocabulary in the FDO record, the record MUST contain
  • an attribute a=(adef,aval) that instantiates attribute definition A, where A applies external vocabulary referencing
  • an attribute b=(bdef,bval) representing a vocabulary term
A and b must fulfill the following requirements:
  • aval is a URL representing a namespace
  • bdef = aval/<term>, i.e., bdef is a URL representing a term under a namespace
  • >bval instantiates bdef.

Validation Rules:

To validate an FDO that contains an attribute a=(adef,aval) applying external vocabulary referencing, it is checked that the namespace aval is valid according to the syntax definition referenced from the attribute definition.
